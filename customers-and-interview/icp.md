## Primary ICP: API Communication Between Backend and Frontend Developers

### 1. Segment

Junior to mid-level backend and frontend developers who are working in small teams (2–6 people), either in professional environments (companies, agencies, freelance collaborations) or in university project teams. These developers regularly collaborate on shared codebases where backend APIs and frontend integrations must stay aligned. They are actively writing and consuming APIs, not just learning about them.

---

### 2. Context

The problem occurs during active development when:

- a backend developer creates or updates an API endpoint
- a frontend developer needs to consume that endpoint
- both sides are working asynchronously or in parallel

This typically happens:

- during feature development cycles
- when new endpoints are introduced
- when existing endpoints change without clear communication

The interaction often takes place across tools such as code editors, GitHub, and messaging platforms.

---

### 3. Primary Pain

Developers do not have a reliable, real-time way to understand the exact structure and behavior of API endpoints. As a result:

- frontend developers must repeatedly ask backend developers for clarification
- backend developers must manually document endpoints multiple times
- inconsistencies occur between implementation and documentation

This leads to:

- delays in feature development
- integration errors
- repeated back-and-forth communication

---

### 4. Current Workaround

Developers currently rely on a combination of:

- Writing manual documentation (Swagger, Postman, README files)
- Sending messages through Messenger, Slack, or other chat tools
- Explaining endpoints verbally or through code snippets
- Debugging integration issues through trial and error

These workarounds are:

- repetitive
- time-consuming
- prone to becoming outdated

---

### 5. Job to Be Done

"When I am working with an API in a team, I want to immediately understand how endpoints work and stay updated when they change, so I can integrate them without delays or repeated clarification."

---

### 6. Success Signal

- Developers no longer need to ask repeated clarification questions
- API changes are visible immediately to all relevant team members
- Frontend integration happens without trial-and-error debugging
- Development flow becomes faster and more predictable

Behavioral change:

- fewer messages asking about endpoints
- less reliance on manual documentation

---

### 7. Where to Find Them

- Your coworkers at Studio Glitch
- Developers in your freelance network
- Classmates working on team projects at KIU
- GitHub collaborators on shared repositories
- Developer communities and group chats you are already part of

You can identify at least 5–10 people immediately who match this profile.

---

### 8. Why You, Why Now

All three team members are developers who have directly experienced this problem. You are actively working in environments where API communication is required, giving you immediate access to real users. This allows:

- fast interview scheduling
- honest, peer-level conversations
- real-time validation within your own workflows

No external access or permission is required to reach this group.

## Secondary ICP: Design Handoff Between Designers and Developers

### 1. Segment

Frontend developers working on websites or web applications based on designs provided by designers, typically in freelance projects, small teams, or agency environments. These developers receive design files (e.g., Figma) that are incomplete in terms of responsive behavior or interaction details.

---

### 2. Context

The problem occurs during the implementation phase when:

- developers translate static design files into working interfaces
- designs only include desktop or partial views
- responsive behavior or interaction logic is not defined

This usually happens:

- at the start of development
- when adapting designs across screen sizes
- when implementing animations or edge cases

---

### 3. Primary Pain

Developers must guess how components should behave because the design does not specify:

- responsive layouts
- component states
- animations or transitions

This leads to:

- additional development time
- inconsistent implementation
- rework after feedback

---

### 4. Current Workaround

Developers typically:

- make assumptions based on experience
- ask designers for clarification
- implement a version and wait for feedback
- iterate multiple times until acceptable

This process is:

- inefficient
- dependent on communication
- prone to misalignment

---

### 5. Job to Be Done

"When I receive a design, I want to clearly understand how every component should behave across different cases, so I can implement it correctly without guesswork or rework."

---

### 6. Success Signal

- Developers implement features without needing clarification
- Fewer revisions are required after implementation
- Designs translate directly into working UI with minimal iteration

Behavioral change:

- fewer back-and-forth messages with designers
- faster implementation cycles

---

### 7. Where to Find Them

- Your own work environment (Studio Glitch)
- Freelance collaborators
- Frontend developers in your network
- Classmates working on UI-based projects

---

### 8. Why You, Why Now

You are a frontend developer who has directly experienced this issue multiple times. You work in environments where design handoff is part of your daily workflow. This gives you:

- direct access to relevant users
- strong understanding of the problem context
- ability to validate insights quickly
